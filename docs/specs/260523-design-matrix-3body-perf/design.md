# Design: Design-matrix construction perf for 3-body SALCs

Status: complete (2026-05-23)

## Summary

Two small, independent rewrites inside `src/Fitting.jl`, each
benchmark-justified by the FeGe 3-body profile. All preserve the
existing public API and produce bit-for-bit identical design matrices
on integration fixtures.

The implementation deliberately stays inside the original
splat-based contraction shape (`coeff_tensor[idx_buf..., mf_idx]`
with per-call `MVector{R - 1, Int}` scratch buffers). Three
alternative rewrites were tried during implementation and rejected
(see "Implementation notes" below).

1. **Replace `searched_pairs::Set{UInt}` with a `Vector{UInt}`** for
   the per-call de-duplication of translated atom tuples. With
   `n_translations` typically O(ntran) ≈ tens, linear scan against a
   contiguous buffer is faster than Julia `Set`'s hash-table
   operations (~9 % torque speedup on the `_light` fixture).
2. **Read spin-direction columns into a stack `SVector{3, Float64}`**
   in three places: the inner SH loops of
   `design_matrix_energy_element` and `calc_∇ₑu!`, and the
   `dir_iatom_svec` construction in `build_design_matrix_torque`.
   The previous form `@views spin_directions[:, atom]` (and the
   variant feeding the inner `Zₗₘ_unsafe` / `∂ᵢZlm_unsafe` calls)
   created a heap-allocated `SubArray` on every read; three direct
   scalar reads into an `SVector` constructor avoid the view
   entirely. Combined ~7 % torque speedup on the `_light` fixture.

## API

Internal signatures stay byte-for-byte identical:

```julia
function calc_∇ₑu!(
    result::MVector{3, Float64},
    cbc::CoupledBases.CoupledBasis_with_coefficient{R},
    atom::Integer,
    spin_directions::AbstractMatrix{<:Real},
    symmetry::Symmetry,
    ws::GradWorkspace,
) where {R}

function design_matrix_energy_element(
    cbc::CoupledBases.CoupledBasis_with_coefficient{R},
    spin_directions::AbstractMatrix{<:Real},
    symmetry::Symmetry,
    ws::EnergyWorkspace,
)::Float64 where {R}
```

`EnergyWorkspace` and `GradWorkspace` field type change only
(`Set{UInt}` → `Vector{UInt}`); no constructor change at the call
sites in `build_design_matrix_energy` / `build_design_matrix_torque`.

## Types and conventions

- Spin-direction layout (`3 × n_atoms`), unit-vector invariant,
  spherical-harmonics convention (`Zₗₘ_unsafe`, `∂ᵢZlm_unsafe`) — all
  untouched.
- `(4π)^(N/2)` per-cluster scaling factor (`_cluster_scaling`) untouched.
- SALC ordering inside `salc_list` and the column order of the design
  matrix are untouched.
- The two refactors do not change any arithmetic: `Vector{UInt}` and
  `Set{UInt}` reach `_atoms_hash_key` with the same values, and the
  `SVector{3, Float64}` carries exactly the three `Float64` entries
  that the `@views` `SubArray` would have read. Design matrices are
  bit-for-bit identical on the integration fixtures.

## Test strategy

- `make test-all` is the gate. Every integration fixture
  (`chain`, `dimer`, `square_lattice`, `fept_tetragonal_2x2x2`,
  `fege_2x2x2`, `febcc_2x2x2_pm`) exercises both the energy and the
  torque design-matrix code paths through end-to-end fits whose stored
  reference values were generated by the pre-change implementation;
  any silent change in `calc_∇ₑu!` or `design_matrix_energy_element`
  output will move those references.
- Bench gate: `bench/bench_b1_3body_fege.jl` on the `_light` fixture,
  4 threads, 5-trial median. Acceptance numbers in `requirements.md`.

## Implementation notes (rejected alternatives)

During implementation we explored three alternative approaches that
turned out worse on the FeGe 3-body benchmark; all are documented
here so a future reader does not re-investigate them.

- **Scalar `Float64` gradient accumulators in `calc_∇ₑu!`.** We
  replaced the `mf_grad_contribution::MVector{3, Float64}` (per
  `mf_idx`) and `grad_result::MVector{3, Float64}` (per `itrans`)
  plus broadcast `.+=` with three scalar `Float64` accumulators
  and explicit per-component `+=`. The rewrite was numerically
  bit-identical (per-component summation order preserved) but
  measured **slightly negative** on torque time, because the
  compiler is already SROA-ing the `MVector` into registers and
  emitting SIMD-friendly 3-wide IR; the scalar form removed that
  pattern. Reverted; the broadcast form is faster.
- **Rank-specialized `if R == 2 / 3 / 4` inline contractions.** We
  wrote out per-axis explicit indexing for each `R`, expecting the
  splat `coeff_tensor[idx_buf..., mf_idx]` to be a major hot spot.
  `Profile.Allocs` revealed that splat indexing of an `MVector{R-1,
  Int}` is in fact alloc-free and well-optimized by the compiler;
  the hand-specialized forms produced equivalent assembly and did
  not measurably improve time or allocations. The current
  splat-based contraction is preserved on grounds of simplicity and
  N-body generality.
- **Generic `CartesianIndices(dims_t)` contraction with workspace-
  backed `Vector{Int}` scratch.** This eliminated the per-call
  `MVector{R - 1, Int}` declarations entirely (allocation count
  dropped 67 %), but the gradient version had to recompute the
  per-site `sh_prod` `2*l_atom + 1` more times than the original
  splat form, costing ~8 % torque wall time. The allocation
  reduction did not translate into a wall-time win, and the
  remaining 112 M allocations all came from the dynamic-dispatch
  closure box on `Vector{CoupledBasis_with_coefficient}` — a
  structural issue out of this spec's scope. We therefore reverted
  to the original splat-based contraction; if dispatch boxing is
  removed in a follow-up spec, this alternative can be revisited.

## Risks and open items

- **`Set{UInt}` vs. `Vector{UInt}` winner depends on
  `n_translations`.** The replacement is A/B-benchmarked on
  `_light` (`n_translations ≈ O(10²)`); the vector wins because the
  de-dup table never exceeds a few dozen entries, well below the
  hash-table break-even. If a future workload pushes
  `n_translations` into the high hundreds the Set form could
  return.
