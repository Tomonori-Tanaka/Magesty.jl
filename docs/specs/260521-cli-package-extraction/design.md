# Design: extract the CLI into a `MagestyCLI` subdirectory package

Status: draft (2026-05-21)

## Summary

`src/CLI.jl` currently holds two unrelated concerns: the `vasp_to_extxyz`
conversion API (plain Julia, no Comonicon) and the Comonicon command tree
(`@cast` / `@main`). Only the second triggers the JET incompatibility. The
split therefore follows that seam:

- The conversion API (`vasp_to_extxyz` and its two private helpers) moves
  into a new Comonicon-free core file, `src/VaspConvert.jl`. It keeps
  using the `VaspIO` and `ExtXYZ` core modules and stays exported from
  `Magesty`.
- The Comonicon command tree moves into a new package, `MagestyCLI`, in
  the `cli/` subdirectory. `MagestyCLI` depends on `Magesty` and
  `Comonicon`; its commands are thin wrappers over `Magesty`'s public API.

The core package then has no `Comonicon` dependency, so `report_package`
no longer macro-expands Comonicon code and `test/jet.jl` reverts to a
plain assertion. `MagestyCLI` is a subdirectory of this repository — one
git history, one CI — not a separate repository; the rationale and the
rejected alternatives (package extension, separate repo) are in the
CLI-package-extraction design note.

## API

No change to the core public API. `vasp_to_extxyz` keeps its signature:

```julia
function vasp_to_extxyz(
    vasprun::AbstractString;
    oszicar::Union{AbstractString, Nothing} = nothing,
    output::Union{AbstractString, Nothing} = nothing,
)::String
```

`MagestyCLI` exposes `MagestyCLI.command_main` and
`MagestyCLI.comonicon_install` (generated by Comonicon). The `Vasp`
command group imports `vasp_to_extxyz` from `Magesty`; the launcher runs
`using MagestyCLI; exit(MagestyCLI.command_main())`.

## Types and conventions

No impact on physics conventions, units, or numerical conventions. Both
the converter and the CLI are pure I/O. No new invariants.

## Test strategy

- `test_ExtXYZ.jl` and `test_VaspIO.jl` use only `Magesty` and are
  unaffected; they continue to run under `make test-unit`.
- `test_vasp_to_extxyz.jl` currently also exercises the CLI through
  `Magesty.command_main`. In M1 that CLI block is removed, leaving the
  `vasp_to_extxyz` API and `output`-keyword tests (which stay valid). The
  equivalent CLI check is re-added in M2 as `MagestyCLI.command_main` in
  `cli/test/runtests.jl`.
- `test/jet.jl` returns to a plain `report_package(Magesty)` assertion;
  `make test-jet` must pass (0 issues), confirming the extraction restored
  JET coverage.
- `cli/test/runtests.jl`: instantiate `MagestyCLI` against the dev-ed
  core, invoke the `extxyz` command on a fixture VASP run, and assert the
  output matches a golden extxyz file. The fixtures are the existing core
  fixtures — `cli/test/runtests.jl` resolves them relative to the
  repository root (`joinpath(@__DIR__, "..", "..", "test", "component",
  "fixtures")`), so no golden file is duplicated into `cli/`.

## Risks and open items

- **Dev workflow.** `MagestyCLI` needs `Magesty` resolvable by path. CI
  and local development use `Pkg.develop(path = "..")`; `cli/Manifest.toml`
  is not committed. The Makefile target encapsulates the instantiate step.
- **Launcher resolution.** The `comonicon_install`-generated `magesty`
  launcher does `using MagestyCLI`, so `MagestyCLI` must be in a resolvable
  environment at run time. This is the standard Comonicon model; the
  installation docs must state it explicitly.
- **Subdirectory name.** `cli/` is the working choice. `apps/MagestyCLI/`
  is the alternative if more sibling packages appear later; deferred —
  renaming a subdirectory later is cheap.
- No numerical results change.
